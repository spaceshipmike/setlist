# Proposal: Digest Generator v2 — Provider Swap, Non-Code Extraction, Large-Spec Support

**Date:** 2026-04-22
**Status:** ACCEPTED — incorporated into spec 0.21 (2026-04-23). See changelog entry for enumerated edits. Retained for historical reasoning.
**Affects:** Section 2.5 (Field Enrichment) or new §2.X, Section 3.1 (Core Capabilities), Section 3.3 (Rules and Logic — digest rules), Section 3.4 (External Connections), `@setlist/cli` `digest` command, possibly `@setlist/core` schema
**Related:**
- `archived/project-essence-digest.md` — the v0.20 proposal that introduced digests for code projects. This extends that work.
- `knowmarks/.fctry/spec-proposals/setlist-digest-as-project-context.md` — KM's parallel cutover proposal. The KM proposal has graceful fallback when a digest is missing, so this setlist-side work does not block it. Both land independently.
**Sequencing:** Can land before, after, or alongside the KM digest cutover. No ordering constraint.

## Why one proposal, three changes

The first run of `setlist digest refresh --all` (2026-04-22) exposed three gaps in the v0.20 generator that share a single root cause — the generator's input and provider assumptions don't match the real portfolio:

1. **Non-code projects were skipped entirely.** 14 of ~25 projects have no `spec.md` / `CLAUDE.md` / `README.md` — they're in `~/Projects/` and carry their content as PDFs, Word docs, plain-text notes.
2. **Large code specs failed outright.** 5 projects failed with generic "fetch failed" errors — all have `.fctry/spec.md` files larger than ~250 KB (fctry at 742 KB, knowmarks at 458 KB, chorus-app at 453 KB, ensemble at 264 KB, plus a transient homelab failure at 51 KB). Source size over ~200 KB consistently blows past the local Qwen3.6-35B model's 128k-token context window, causing MLX server crashes and connection drops.
3. **Local-only generator is slow and fragile.** A full portfolio refresh takes ~30 minutes of serialized MLX thinking-mode generation; transient failures are common; the M4 Pro has to be awake and un-OOM'd; offload for hobby experiments is blocked on digest availability.

All three resolve cleanly if the generator gains (a) a provider knob that defaults to a hosted model with a 1M context window, and (b) an extraction fallback for non-text project sources. The cost is pennies per run. This proposal does both together because skimming one and deferring the other would leave the generator broken for most of the portfolio either way.

## Problem

Setlist 0.20 introduced per-project essence digests generated by `@setlist/cli digest refresh`. The generator works for **code projects** — it reads `.fctry/spec.md`, `CLAUDE.md`, or `README.md` in that order, extracts a `spec-version` from YAML frontmatter, and calls the home local LLM endpoint for the summary.

**Non-code projects have no such input.** Projects in `~/Projects/` — time-bound work without a git repo per the root CLAUDE.md convention — carry their content as heterogeneous documents: PDFs, Word docs, plain-text notes, screenshots, Pages files, exported Google Docs. They often have no `.fctry/spec.md`, no `CLAUDE.md`, no `README.md`, and no `spec-version` anywhere.

A run of `setlist digest refresh --all` on the current portfolio (2026-04-22) demonstrates the gap. Of the projects that were skipped:

- `41m-apartment-renovation` — non-code, no spec/CLAUDE/README
- `bss-history-project` — non-code, no spec-version
- `bss-speaker-series` — non-code, no spec/CLAUDE/README
- `fam-austin-gp-trip`, `fam-colorado-trip`, `fam-emergency-info-packet`, `fam-estate-planning` — all non-code, all skipped
- `fig-gold-site`, `health-fitness-plan` — same

These projects are legitimate entries in the registry. Consumers asking "what is project X about?" — the orchestrator ref scanner, Knowmarks project-theme scoring, any future cross-project query — get nothing useful for this entire class of projects.

## Design decisions

### 1. Default provider: Gemini 2.5 Flash-Lite via OpenRouter; local MLX as fallback

The generator gains a provider knob with three values: `openrouter-flash-lite` (default), `openrouter-flash` (opt-in for higher-quality work), `local-mlx` (fallback / offline / privacy-sensitive runs). The selection is a single env var, `SETLIST_DIGEST_PROVIDER`, read at `@setlist/cli` startup.

**Why Flash-Lite as default:**

- **1M-token context window** eliminates the oversized-spec failure mode entirely. fctry's 742 KB spec (~185k tokens) fits with 5x headroom. No input truncation, no chunking, no retry-on-connection-drop plumbing needed.
- **~2s per call** vs ~2:44 for local Qwen3.6-35B thinking-mode generation. A full portfolio refresh drops from ~30 minutes of serialized generation to ~30 seconds.
- **Cost is rounding-error.** Current portfolio of ~11 code specs totals ~611k input tokens; at Flash-Lite's $0.10/M input + $0.40/M output, a full refresh costs **~$0.07** ($0.035 via the 24h Batch API). A worst-case fctry-only refresh is ~$0.02.
- **Quality fit.** Summarization under strict scope constraints ("current-state only, name unbuilt sections, no marketing voice") is exactly what Flash-Lite is tuned for — instruction following over raw capability.
- **Reliability.** No dependency on the home Mini being awake, un-OOM'd, or on-network. The Mini can be repurposed for hobby experiments without breaking the digest pipeline.

**Why OpenRouter as the provider surface:**

- Already the portfolio's established LLM provider (Knowmarks' default, per its CLAUDE.md). One vendor, one key style, one billing dashboard.
- OpenAI-compatible API — the existing generator in `packages/cli/src/digest.ts:78–117` works unchanged except for base URL, model name, and a bearer-token header.
- Supports cost-attribution headers (`HTTP-Referer`, `X-Title`) that surface on the OpenRouter dashboard. Per the LLM-cost-isolation preference, setlist gets its own OpenRouter API key and tags every call as `X-Title: setlist-digest-generator`.
- Provider-fallback routing available if a specific upstream goes down; cost impact is tiny.

**API key location.** The key lives in 1Password (`op://Dev/openrouter-setlist/credential`) per the root CLAUDE.md secrets convention, resolved at runtime via `op read` in the shell environment that invokes `setlist digest refresh`. Envs without a key fall through to `local-mlx` automatically (log one line, do not error).

**Local MLX as fallback:** retained, not removed. When `SETLIST_DIGEST_PROVIDER=local-mlx` is set explicitly, or when the OpenRouter key is missing, or when Flash-Lite returns a 5xx twice in a row, the generator falls back to the existing `http://m4-pro.local:8000/v1` path. The local path keeps the same size-limit constraint (input under ~100k tokens) and gains a 3-retry exponential-backoff wrapper to handle the homelab-style transient failures.

**Input-size handling becomes provider-aware.** Flash-Lite/Flash: no truncation, full spec sent. Local-MLX: head-truncate at 400 000 chars (~100k tokens) with a log line when it fires. One function, two branches.

### 2. Extraction is a generator-side concern, not a setlist-core concern

The existing proposal explicitly placed generation outside `@setlist/core`:

> *Setlist stores digests and tracks their provenance … It does not generate them.*

That posture holds here too. The MCP surface does not change. The schema does not change. All new behavior lives in `@setlist/cli digest refresh` and its helpers. The digest that lands in `project_digests` looks exactly the same as a code-project digest — same shape, same fields, same ceiling.

### 3. A new extraction step, feeding the existing LLM pipeline

When the current `readProjectSource` helper finds no `spec.md` / `CLAUDE.md` / `README.md`, the generator does not give up — it falls through to an extraction path:

1. Walk the project's root directory (from `project_paths`), one level deep for v1, collecting supported document files.
2. Extract markdown text from each file using a format-specific extractor.
3. Concatenate extracted text in a deterministic order (alphabetical by path).
4. Feed the concatenated blob to the same LLM with the same system prompt.
5. Write the result via `refreshProjectDigest` exactly as today.

The LLM sees clean markdown whether its upstream source was a `spec.md` or an `.docx` — the generator normalizes the input.

### 4. Tool choice for non-code extraction: docling via Python subprocess

Three candidates were weighed:

| Tool | Coverage | Ergonomics from TS | Output quality |
|------|----------|---------------------|-----------------|
| **docling** (IBM Research, Python) | PDF, DOCX, PPTX, XLSX, HTML, images (OCR) | Requires Python subprocess | High — layout-aware markdown |
| **kreuzberg** (Python, used by Knowmarks) | PDF, Office, HTML, OCR | Requires Python subprocess | High |
| **pandoc** (binary) | DOCX, HTML, RTF, EPUB — weak on PDFs | Easy `exec` | Mid — PDF support requires pdftotext fallback |

Recommendation: **docling**, called via a small Python subprocess helper shipped with `@setlist/cli`. Rationale:

- Best all-round coverage including PDFs with layout awareness
- Active upstream (IBM, regular releases through 2026)
- Outputs markdown directly — no post-processing needed to get to the LLM input shape
- Python subprocess is not architecturally foreign to setlist: the CLI already calls the local LLM over HTTP, so "shell out to external tooling" is an established boundary

The subprocess shape is a single script, `packages/cli/python/extract.py`, invoked as:

```bash
python3 extract.py <file-path> --format markdown
```

with stdout as the extracted markdown. Install path is the user's existing Python (same path that hosts `docling`); the `docling` package is a user-installed dependency, documented in the setlist bootstrap docs.

Pure-markdown inputs (`.md`, `.txt`, `.html`) are read directly in TypeScript — no subprocess needed for those formats.

### 5. Staleness: file-tree hash as the `spec_version`

Code projects carry a `spec-version` string in YAML frontmatter; the digest's `spec_version` column stores that exact value, and staleness is a simple equality check.

Non-code projects have no frontmatter. Proposal: compute a deterministic **file-tree hash** across the supported documents in the project directory (one level deep, alphabetical, each entry as `"<relative_path>:<mtime>:<size>"`, hashed with sha256, truncated to 16 hex chars). Store that hash as the digest's `spec_version`.

On subsequent reads, `get_project_digest` computes `stale` by recomputing the hash and comparing. Any document added, removed, edited, moved, or resized changes the hash and flips `stale: true`. No TTL, no wall-clock drift, same deterministic signal.

The `spec_version` column type (TEXT) already admits this; no schema change is required. Consumers treat it as an opaque identifier — they already check `stored != current` regardless of format.

### 6. Generator version bump: encode provider + extractor in `producer`

The existing `producer` column carries `local-<model-tag>`. With provider selection plus optional document extraction, it grows two dimensions. Proposed format:

```
<provider-tag>[+<extractor-tag>]
```

Examples:

- `openrouter-google/gemini-2.5-flash-lite` — Flash-Lite, code spec source, no extraction
- `openrouter-google/gemini-2.5-flash-lite+docling-2.18` — Flash-Lite, non-code project, docling extracted inputs
- `local-mlx-community/Qwen3.6-35B-A3B-8bit+docling-2.18` — local fallback with extraction
- `manual` — human-written, unchanged

Consumers that care about provenance (future audit tooling, or a "regenerate this digest against the newer model" affordance) have enough signal. Consumers that only care about the text ignore the tag.

### 7. Project type inference

The generator should not require the user to mark projects as "code" vs "non-code." Inference is straightforward:

- If `readProjectSource` returns text **and** a `spec-version` is extractable → treat as a code project; use spec-version as the version stamp
- Otherwise → treat as a non-code project; extract documents if any are found; use file-tree hash as the version stamp
- If no supported documents exist at all → skip with a clearer message than today ("no spec.md / CLAUDE.md / README.md / supported documents found, skipped")

Same command (`setlist digest refresh --all`), same output shape, richer coverage.

## CLI interface

No new commands. Existing invocations gain coverage:

```bash
setlist digest refresh 41m-apartment-renovation
setlist digest refresh --all
setlist digest refresh --stale
```

Output adopts a consistent form regardless of provider, source path, or extraction step:

```
✓ 41m-apartment-renovation — 612 tokens, fs-hash a3f9c2b8e74d1062, producer openrouter-google/gemini-2.5-flash-lite+docling-2.18
✓ fctry — 1089 tokens, spec 0.16, producer openrouter-google/gemini-2.5-flash-lite
✓ homelab — 712 tokens, spec 0.2, producer local-mlx-community/Qwen3.6-35B-A3B-8bit (fallback: OpenRouter key missing)
```

The `producer` tag conveys the full generation path: provider + model + optional extractor. Stamps to the left (`spec X.Y` or `fs-hash …`) convey whether the project was treated as code or non-code.

## Consumer impact

None on the read path. `get_project_digest` and `get_project_digests` return the same response shape they do today. `stale` still means "the stored version stamp differs from the current one" — how that stamp is computed (spec-version vs fs-hash) stays internal to setlist-core's project-version resolver.

Three downstream improvements flow naturally:

- **Knowmarks' in-flight cutover** gets digests for every active project, including non-code ones its project-theme scoring previously had no handle on.
- **Orchestrator ref scanning** (R2 step) now receives useful text across the entire portfolio.
- **Refresh latency** drops from ~30 minutes to ~30 seconds for a full run, unblocking a `setlist digest refresh --stale` cron without a user-visible cost hit.

## Observability

Because LLM spend now flows through OpenRouter, the generator tags every call with:

- `X-Title: setlist-digest-generator` — surfaces as the app name on OpenRouter's dashboard
- `HTTP-Referer: https://github.com/mikelebowitz/setlist` (or the local-dev fallback) — app-identity URL
- Model and token counts per call logged locally to `@setlist/cli` stderr at INFO level (off by default; `SETLIST_DIGEST_VERBOSE=1` to enable)

Per the LLM cost-isolation preference, setlist's OpenRouter key is dedicated — not shared with Knowmarks, Chorus, or other projects. One project, one key, one row on the billing dashboard.

## Schema impact

None required for v1. The existing `project_digests` table carries everything needed.

Optional future additions (not in this proposal):

- `source_type` column on `project_digests` — `"spec-text"` vs `"document-extraction"`. Would help consumers distinguish if behavior ever diverges. YAGNI until a consumer asks for it.
- Per-project `version_source` preference stored on `projects` — if a user ever wants to override the inference rule. YAGNI.

## Scenario sketches

For the implementing cycle to write out fully:

1. **Large code spec via Flash-Lite.** Refresh on fctry (742 KB spec, ~185k tokens) completes in under 10 seconds end-to-end, producing a digest within the 500–1200 token range, `producer` tag is `openrouter-google/gemini-2.5-flash-lite`, no retries recorded.
2. **Provider fallback when key is missing.** Refresh with `SETLIST_OPENROUTER_API_KEY` unset produces a digest via the local MLX endpoint, one INFO log line names the fallback, output shows the local producer tag.
3. **Explicit local override.** `SETLIST_DIGEST_PROVIDER=local-mlx setlist digest refresh setlist` uses the local endpoint even with OpenRouter configured. Useful for privacy-sensitive runs or offline work.
4. **Non-code project with one PDF.** Refresh produces a digest via Flash-Lite, `spec_version` is a 16-char hex hash, `producer` tag includes the docling component, token count is within target.
5. **Non-code project with mixed formats.** PDF + Word + a plain-text note in one directory. Concatenated text preserves order; LLM produces a coherent digest that references content from all three sources.
6. **File edit flips stale.** After a successful refresh, edit any supported file (touch mtime) and call `get_project_digest` — `stale: true`. Refresh — new hash stored, `stale: false`.
7. **No supported documents.** A non-code project directory with only `.DS_Store` and `.git` → skipped with a clear message, no error.
8. **Mixed project.** A project directory that has both a `spec.md` with `spec-version` and stray PDFs → code-project path wins; PDFs ignored for digest purposes (spec.md is authoritative).
9. **Docling not installed.** Subprocess returns non-zero → generator falls back to text-only extraction for the non-PDF/non-Office formats present; if nothing survives, project is skipped with a message suggesting docling installation.
10. **OpenRouter 5xx retry.** Provider returns 503 twice in a row on a single call → third attempt either succeeds or falls through to local MLX with a log line; refresh does not crash the batch.
11. **Cost tagging.** Every Flash-Lite call includes `X-Title: setlist-digest-generator`; OpenRouter dashboard shows the portfolio refresh as a single attributable line item.

## Out of scope

- **Automatic watching / refresh-on-save.** The refresh cadence remains explicit (`--all`, `--stale`, or per-project). A file-change watcher is a separate infrastructure concern.
- **Digest kinds other than `essence`.** The single-kind v1 posture from the original digest proposal holds.
- **Semantic chunking of input.** v1 concatenates naively. If large non-code projects become common, chunking + map-reduce over the LLM becomes worthwhile — deferred.
- **Image-heavy non-code projects.** The docling OCR path handles scanned PDFs. Standalone image files are deferred; the VL model variant is not deployed on the home LLM endpoint.
- **Cloud document sources.** Google Docs, Notion pages, shared drives — all would need connector infrastructure. Setlist's digest generator reads filesystem only; users who want a cloud doc represented in a digest should export it locally first. (Knowmarks is the right home for cloud connectors; setlist should not grow them.)

## Open questions for the evolve conversation

1. **Fallback trigger policy.** When should the generator fall back from OpenRouter to local MLX? Options: (a) only when the API key is missing at startup, (b) also on two consecutive 5xx responses, (c) also on rate-limit 429s, (d) also on cost ceiling hit. Recommended: (a) + (b) in v1; add (c) and (d) if they come up in practice.
2. **Cost ceiling / circuit breaker.** Even at pennies per full refresh, an accidental loop could rack up. Recommended: a per-invocation hard cap (e.g. $1.00) enforced client-side by tracking total input+output tokens against a known rate. If hit, the remaining projects in the batch fall back to local MLX or skip.
3. **Model choice default.** Flash-Lite is the recommended default for this task. Should the generator pick Flash (non-Lite) automatically for certain projects, or always require the user to opt up? Recommended: always Flash-Lite by default; `SETLIST_DIGEST_MODEL=google/gemini-2.5-flash` opt-in for cases where a re-read shows Flash-Lite produced shallow output.
4. **Depth of directory walk.** v1 proposal: one level deep (documents at the project root). Deeper walk is straightforward to add but increases the risk of pulling in noise (e.g. an archived subfolder). Recommended: one level in v1; add `--depth N` flag if a project demonstrates need.
5. **Docling version pinning.** Pinning a specific docling version in setlist's documented bootstrap makes generator output reproducible across machines but adds maintenance burden. Recommended: document a tested version in the bootstrap doc, do not hard-require it at runtime.
6. **What counts as a "supported document."** v1 proposal: `.md`, `.txt`, `.html` (native TS read), `.pdf`, `.docx`, `.pptx`, `.xlsx` (docling subprocess). Recommended: ship that set; expand only when a real project fails for a specific format.
7. **Input-size ceiling on the local path.** When the generator falls back to local MLX, the 128k context window returns as a constraint. Head-truncate at 400 000 chars (~100k tokens) vs head+tail split vs map-reduce summarize. Recommended: head-truncate in v1 with a log line; revisit if real projects show important tail content being lost.

## Convergence sketch

- `@setlist/cli/digest.ts` gains a **provider abstraction**: `generateDigest` branches on `SETLIST_DIGEST_PROVIDER`, talks to OpenRouter by default (`https://openrouter.ai/api/v1/chat/completions` with bearer key and cost-attribution headers), falls through to the existing local MLX path when unconfigured or on repeated failure. Input truncation only applies on the local branch.
- `@setlist/cli` gains `packages/cli/python/extract.py` (a ~30-line wrapper around docling's Python API) plus a TS-side orchestrator that walks the project directory, decides per-file whether to read natively or shell out, concatenates results, and feeds the normalized text to `generateDigest`.
- `@setlist/core` gains a `computeProjectVersion(projectName)` helper that reads YAML spec-version if present, otherwise computes the file-tree hash. Used by the generator and by `get_project_digest` for staleness checks.
- No schema change. No MCP surface change.
- Scenarios 11 new (sketched above).
- Spec section touches:
  - §2.5 Field Enrichment (or new §2.X) describing the generator's three-branch input flow (code spec → non-code extraction → manual/skipped) and the provider selection
  - §3.1 Core Capabilities adding "document extraction for non-code digests" and "hosted-provider default with local fallback"
  - §3.3 Rules and Logic: digest rules gain an input-size rule that is provider-aware
  - §3.4 External Connections: OpenRouter listed as an outbound dependency of `@setlist/cli` (not of `@setlist/core` or the MCP server, which stay pure-data)
  - Synopsis `patterns` list gains `hosted-digest-generation-with-local-fallback`, `document-extraction-for-non-code-digests`, and `project-tagged-llm-cost-attribution`
- 1Password: new `Dev/openrouter-setlist` item holding a dedicated OpenRouter key. Bootstrap doc updated with the one-line `op read` invocation.

Target spec version: 0.20 → 0.21.
